HTML Document Principles

이 프로젝트의 기록 문서는 Markdown 벽보다 브라우저에서 바로 열 수 있는 HTML artifact를 우선한다. 기준은 Thariq Shihipar의 The unreasonable effectiveness of HTML에서 가져온다.

우리가 받아들이는 철학

  • 한 파일을 열면 맥락이 보여야 한다. 도구, 빌드, 프레임워크 없이 브라우저에서 바로 읽혀야 한다.
  • 순차적인 긴 글보다 구조를 먼저 보여준다. 표, 목록, 단계, 다이어그램으로 비교와 결정을 스캔 가능하게 만든다.
  • 문서는 작업을 대체하지 않고 다음 작업을 가능하게 한다. 결정 문서는 구현자와 발표자가 바로 행동할 수 있어야 한다.
  • 시각화는 장식이 아니라 압축이다. 다이어그램, 표, 접힘 영역은 정보를 더 빨리 파악하게 할 때만 쓴다.
  • HTML artifact를 우선한다. 문서는 브라우저에서 바로 열려야 하며, CSS는 읽기 좋은 표면을 만드는 보조 수단으로만 쓴다.

작성 규칙

규칙 이유 예시
HTML 구조를 먼저 쓴다. 스타일이 빠져도 제목, 표, 목록, 링크만으로 문서가 읽혀야 하기 때문이다. 결정은 table, 절차는 ol, 관계는 다이어그램 또는 목록.
표는 결정을 요약할 때 쓴다. 모델, 데이터, risk, next step을 한눈에 비교할 수 있다. ADR 요약, dataset count, runbook command.
다이어그램은 inline SVG 또는 단계 목록으로 남긴다. 이미지 파일 없이도 흐름을 보여주고, 텍스트 fallback도 유지한다. 입력 → 모델 → 결과 흐름도.
모든 문서는 다음 행동을 포함한다. 좋은 기록은 읽고 끝나는 것이 아니라 이어서 실행할 수 있어야 한다. 서버 명령, 검증 항목, 남은 질문.
검증된 사실과 가정을 구분한다. 모델 성능, dataset 수량, 서버 상태는 바뀔 수 있기 때문이다. "내부 split 기준", "아직 검증 전", "pending decision".

언제 CSS나 JS를 허용할 수 있는가

기본값은 HTML-first다. CSS는 문서 허브처럼 팀원이 자주 보는 화면의 가독성을 높일 때 허용한다. JavaScript는 클릭 가능한 prototype, slide deck, interactive evaluator처럼 상호작용 자체가 문서의 목적일 때만 쓴다. 이때도 build step과 무거운 dependency는 피한다.
